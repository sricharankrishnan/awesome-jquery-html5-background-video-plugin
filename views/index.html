Jekyll Lava Boiler Plate

Upgraded with Jekyll Version 4.0 and comes with a package.json file for running terminal level commands.

Hello JLBP

Jekyll Lava Boiler Plate (or just JLBP), is a project built to help developers get started with a jekyll build environment for their static site projects. It comes pre-built with certain features that can be useful for production level sites.

JLBP is being served right now with Jekyll Version 4.0. Apart from that, it has been initiated with npm init, thus comes with a package.json file giving you possibilities of expanding it further (thought about using Grunt JS with Jekyll?).

Getting Started

Go to the github source page by clicking on the link below. Then get started with JLBP by reading the instructions below.

JLBP is not an opiniated Jekyll Build. The idea is to provide a platform and a build environment for developers. If you understand Jekyll and JLBP, perhaps the possibilities are endless.

JLBP assumes that you have prior knowledge of how to work with jekyll. In case you don't, please click on the link below that takes you to their website. You also need to have jekyll locally installed in your system (with version 4.0) along with ruby.

Do checkout the installation process with the jekyll website. If you already have these details ready, you can proceed to the next step.

Download The JLBP Scaffolding

JBLP can be downloaded from its github remote. You can start off by typing git clone git@github.com:sricharankrishnan/jekyll-lava-boilerplate.git [./your-project-directory-name] to get started.

Allow the download process to complete and once this is done, proceed into your project directory name with cd ./your-project-name and follow along with the rest of the details below.

Remove These Files and Settings

Run terminal command npm run cleanup post the download (possible because JLBP comes with a package.json file which you will see in the next section). This command helps to remove certain unwanted files and remotes. The list of details that are being removed from this scaffolding are listing below for your information:

  • The Github Remote: This project has a github remote from which you may have clone the scaffolding. Please perform git remote remove origin. You can set your own respective github remotes for your projects.
  • Unwanted Files: CODE_OF_CONDUCT.md, CONTRIBUTING.md, LICENSE, README.md, googleac250e85da349e6d.html - are all files that may not be suitable for your project. These files are particularly used for hosting this project with Github/Google search console.
  • Cannonical Link Tag (Requires Manual Removal!) Head to the _layouts/default.html file, remove <link href="https://jekyll-lava-boilerplater.netlify.com/" rel="cannonical"/> as this is also applicable for the JLBP project.

With all of these details removed, lets now look at starting the server on your local machine.

Package.json and Server Start

JLBP comes with a package.json file. It has a couple of commands that you can use (in the scripts property) that makes it easy to run the server. They are:

  • npm run dev This is the standard and basic command that you can run on the terminal to start the jekyll server. A watch is kept on all files in your projects and will automatically update the build.
  • npm run devhost Similar to the previous command but the difference being that you can see your jekyll website on a another device as long as you are on the same Wifi connection. The availability of this feature is purely dependent on Jekyll itself and is not propreitary to JLBP.
  • npm run hotreload Helps to run the server like npm run dev but will auto refresh your browser once you save changes in your project file(s). The availability of this feature is purely dependent on Jekyll itself and is not propreitary to JLBP.
  • npm run build Performs the required build process. This will help to generate the required static files, once ready, you can then check out the _site directory for the folders and files that you can use to place it in a server or hosting area for your live website.

Once you've removed all unwanted files and remote links mentioned from the previous section, you should be able to start the server with any one of the commands above. When you do open your browser with http://localhost:4000/, you should be able to see this exact same page.

With that you're set. Lets now see some of the features of JBLP in the following section

JLBP comes with a couple of features that help with the development process. Minifying HTML and Sass file are some of the things that you can expect.

Assets Directory

This directory has been created with the idea of housing all static assets. These would normally be files that are images, videos, stylesheets, javascript files and more. If you have a look at the directory (in javascript for example), you'd see that it even houses jquery and bootstrap.

Consider this to be a good place to store vendor requirements and depenedencies as well.

View Directory

All page related html files originate from this folder. This place represents the main body of a particular page. From here, we can call on jekyll features like {% include some-partial-file.html %} ifyou like to render a partial file into your main body.

When you run the build command all page content files are finally built from the files housed in this directory.

Rel Base

Rel Base is a feature that was picked up from here. Its very useful in providing the much needed relative path builds to connect your file requirements with the main html file. Would encourage that you read through that blog for more information.

The Compress HTML File

The compress.html file located in./ _layouts directory helps in compressing the HTML file that is finally being rendered on the browser. Note that this is actually achieved by initiailizing ./_layouts/default.html file with the needed front matter on top, otherwise it won't work.

A Note on the Directory Structure

From the initial days, its been found necessary to have a good directory structure to keep things in place. Thus as a result, directories like _sass for example, have certain files split up into folders inside of it. The modules directory houses the _variables.scss and _mixins.scss files. You can open these up to see their contents, add new variables or mixins. Perhaps you may even want to add a file of a similar nature or purpose in this folder.

vendor-components refers to directories or folders that have third party files as dependencies. The _sass directory houses bootstrap inside this and assets directory houses jquery and others. Again, you could choose to have files of similar nature (3rd party sass files) and keep them here for importing into your page files.

The common directory is kept to house all common styles for the entire web site/static web app. You can add additional or new styles for your app's navigation, footer section and all other common styles applicable. These can then be imported. Take a look at ./assets/css/home-page.scss file to see what is being done to get an understanding.

What about the others you ask?

All other folders and files like 404.html, _config.yml, Gemfile, Gemfile.lock, _includes, _layouts, _sass, _site are all a part of the original build from Jekyll. Like mentioned earlier, you'd need to have some understanding of how jekyll works in order to get this parts as well.
